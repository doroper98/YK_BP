# BP_WORKFLOW — AI 협업 개발 워크플로우

> 요구사항 접수부터 배포까지, 사람과 AI가 어떻게 역할을 나누고 협업하는지 정의합니다.

## 전체 흐름

```
[사람] 요구사항 제시
    ↓
[AI] 요구사항 분석 → GOAL.md 반영
    ↓
[AI] 구현 (코드 작성)
    ↓
[AI] CLI Gate 자동 검증 (tsc + build)
    ↓
[사람] 수동 검증 (UI 확인, 피드백)
    ↓
[사람] 버그 리포트 / 추가 요구사항
    ↓
[AI] 수정 → CLI Gate → 재검증
    ↓  (반복)
[AI] DEVLOG 기록 + 버전 업
    ↓
[AI] 빌드 + 배포 (exe/패키지)
```

## 역할 분담

| 단계 | 사람 | AI |
|------|------|-----|
| 기획 | 요구사항 한국어로 자유롭게 제시 | 구조화된 REQ-* / SC-* 로 변환 |
| 설계 | 승인/거부 | 아키텍처 제안, 파일 구조 수립 |
| 구현 | — | 코드 작성, CLI Gate 통과 |
| 1차 검증 | UI 확인, 스크린샷 피드백 | CLI Gate (tsc, build, lint) |
| 버그 리포트 | 자연어로 현상 설명 | 원인 분석 → 수정 → 재검증 |
| 추가 요구 | 자연어로 새 기능 요청 | 기존 코드 분석 → 구현 → 검증 |
| 버전 관리 | — | package.json + DEVLOG 원자적 업데이트 |
| 배포 | 승인 | electron-builder → exe 생성 |

## 요구사항 전달 Best Practice

### 좋은 요구사항 (사람 → AI)

```
"회의 노트에서 텍스트를 드래그해서 선택하려고 하면 노드가 움직이는 버그가 있어.
그거 수정하고, 회의 노드는 왜 세로 스크롤이 무조건 생기는 거야?
그리고 이미지를 붙여넣으면 사이즈 조절이 됐으면 좋겠어."
```

**왜 좋은가:**
1. 현상을 구체적으로 설명 (드래그 → 노드 움직임)
2. 여러 요청을 한 번에 (AI가 관련성 파악 가능)
3. 기대 결과를 명시 (사이즈 조절이 됐으면)

### 버그 리포트 (사람 → AI)

```
"여전히 해상도가 거지 같은데?"
```

**이것도 유효한 리포트:**
- AI가 이전 맥락에서 어떤 수정을 했는지 알고 있음
- "여전히" = 이전 수정이 효과 없었다는 피드백
- AI는 다른 접근법으로 재시도해야 함

### 추가 기능 요청 (사람 → AI)

```
"지금 follow up 버튼 클릭했을 때 뜨는 창의 테마가 아주 좋아.
그걸 git 버튼이랑 기능 버튼들에게 다 적용해."
```

**왜 효과적인가:**
- 기준점을 명확히 제시 (FollowUp 패널의 테마)
- 적용 범위를 지정 (git, 기능 버튼 전체)
- AI가 기준 패널의 스타일을 분석 → 나머지에 일괄 적용

## 단계별 상세 절차

### Phase 1: 프로젝트 초기화
```
1. CLAUDE.md 작성 (→ BP_CLAUDE_MD.md 참조)
2. GOAL.md 작성 (→ BP_REQUIREMENTS.md 참조)
3. WORKFLOWS.md 작성
4. docs_canonical/ 4종 작성
5. 프로젝트 스캐폴딩 (package.json, tsconfig, etc.)
6. Git 초기화 + 첫 커밋
```

### Phase 2: 구현 사이클 (반복)
```
1. [사람] 요구사항/버그 전달
2. [AI] GOAL.md의 REQ/SC 확인
3. [AI] 코드 구현
4. [AI] CLI Gate: tsc --noEmit && npm run build
5. [AI] 성공 → DEVLOG EXP-{NNN} 기록
6.       실패 → 원인 분석 → 3번으로
7. [사람] 앱 실행 → 수동 검증
8. [사람] OK → 다음 요구사항 / NG → 버그 리포트
```

### Phase 3: 버전 릴리스
```
1. [AI] 모든 SC PASS 확인
2. [AI] package.json 버전 업
3. [AI] DEVLOG 개선 로그 기록
4. [AI] git commit + push
5. [AI] electron-builder → exe 생성
6. [사람] 설치 테스트 → 승인
```

## 세션 간 컨텍스트 유지

AI 세션이 끊어져도 다음 세션에서 이어갈 수 있도록:

1. **CLAUDE.md** — 실행 규칙, 정규 문서 경로 (매 세션 자동 로드)
2. **DEVLOG.md** — 지금까지 한 일, 버그, 교훈 (AI가 참조)
3. **GOAL.md** — 남은 요구사항, 성공 기준 현황
4. **Git log** — 커밋 히스토리로 진행 상황 파악
5. **package.json version** — 현재 버전으로 진행 단계 유추

## Anti-Pattern (피해야 할 것)

| Anti-Pattern | 왜 나쁜가 | 대안 |
|-------------|-----------|------|
| 요구사항을 너무 추상적으로 | AI가 방향을 잡지 못함 | 현상 + 기대결과 명시 |
| CLI Gate 생략 | 빌드 안 되는 코드 누적 | 매 단계 필수 실행 |
| DEVLOG 미기록 | 다음 세션에서 맥락 소실 | 커밋마다 기록 |
| 한 세션에 너무 많은 변경 | 컨텍스트 초과 → 품질 저하 | 기능 단위로 세션 분리 |
| 피드백 없이 "계속 해" | AI가 잘못된 방향으로 진행 | 중간중간 스크린샷 확인 |

## TCC 프로젝트 실적

- 4개 Phase, 29회 실험, **100% 성공률**
- 8개 버그 발견 → 전부 해결
- v0.1.0 → v1.0.0까지 20+ 버전 점진적 개선
- 사람은 요구사항/피드백만 제공, AI가 구현/검증/기록 전체 수행
