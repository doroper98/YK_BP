# BP_REQUIREMENTS — 요구사항 정의 & AI 전달 방법

> 사람이 자연어로 말한 요구사항을 구조화된 형식으로 변환하고,
> 성공 기준(SC)으로 검증 가능하게 만드는 체계입니다.

## GOAL.md 템플릿

```markdown
# {프로젝트명} — Project Goal

## 1. Objective (목적)
{프로젝트가 해결하려는 문제, 1-3문장}

## 2. Requirements (요구사항)

### 기능 요구사항 (Functional)

| ID | 분류 | 설명 | 우선순위 |
|----|------|------|----------|
| REQ-AUTH-001 | 인증 | 비밀번호 기반 로컬 로그인 | P0 |
| REQ-UI-001 | UI | 3분할 레이아웃 (트리+캔버스+패널) | P0 |
| REQ-UI-002 | UI | 캔버스 줌/팬 (25-400%) | P1 |
| ... | ... | ... | ... |

### 비기능 요구사항 (Non-Functional)

| ID | 설명 |
|----|------|
| NFR-001 | 앱 시작 ≤ 5초 |
| NFR-002 | 60fps 캔버스 렌더링 |
| NFR-003 | 오프라인 동작 (네트워크 불필요) |

## 3. Success Criteria (성공 기준)

| SC | 설명 | 검증 방법 | REQ |
|----|------|-----------|-----|
| SC-01 | 앱이 5초 내 실행 | 수동 측정 | NFR-001 |
| SC-02 | 로그인 성공/실패/잠금 동작 | 수동 테스트 | REQ-AUTH-001 |
| SC-03 | 3분할 레이아웃 표시 | 수동 확인 | REQ-UI-001 |
| ... | ... | ... | ... |

## 4. Phases (단계)

### Phase 1: {단계명}
- S1.1: {단계 설명} → SC-01
- S1.2: {단계 설명} → SC-02, SC-03
- ...

### Phase 2: {단계명}
- S2.1: ...
```

## 요구사항 ID 체계

```
REQ-{Domain}-{NNN}
      │         │
      │         └── 일련번호 (001~)
      └── 도메인 (AUTH, UI, DATA, GIT, AI, AUDIO, ...)

NFR-{NNN}
     └── 일련번호

SC-{NN}
    └── 일련번호
```

## 사람 → AI 요구사항 전달 패턴

### 패턴 1: 새 기능 요청
```
사람: "이미지를 붙여넣으면 사이즈 조절이 됐으면 좋겠어"

AI가 해야 할 일:
1. GOAL.md에 REQ 추가 (REQ-UI-NNN)
2. SC 추가 (SC-NN: 이미지 붙여넣기 후 드래그로 리사이즈 가능)
3. 구현
4. CLI Gate
5. DEVLOG EXP 기록
```

### 패턴 2: 버그 수정 요청
```
사람: "텍스트 드래그하면 노드가 움직여"

AI가 해야 할 일:
1. DEVLOG BUG-NNN 등록
2. 원인 분석 (이벤트 전파 문제)
3. 수정
4. CLI Gate
5. BUG 상태 → RESOLVED
```

### 패턴 3: 스타일/테마 변경
```
사람: "follow up 창의 테마가 좋아. 다른 패널들에도 적용해"

AI가 해야 할 일:
1. 기준 패널 스타일 분석
2. 나머지 패널에 일괄 적용
3. DEVLOG 개선 로그 기록
```

### 패턴 4: 이전 수정이 효과 없음
```
사람: "여전히 해상도가 거지 같은데?"

AI가 해야 할 일:
1. 이전 접근법이 실패했음을 인지
2. 근본 원인 재분석
3. 다른 접근법으로 수정
4. DEVLOG Lessons Learned에 기록
```

## 우선순위 체계

| 등급 | 의미 | 처리 |
|------|------|------|
| P0 | 필수 (없으면 릴리스 불가) | 즉시 구현 |
| P1 | 중요 (있으면 크게 개선) | Phase 내 구현 |
| P2 | 선호 (있으면 좋음) | 여유 시 구현 |
| P3 | 향후 고려 | 백로그 |

## 성공 기준(SC) 작성 규칙

### 검증 가능하게 쓸 것
```
# Bad
SC-05: 노드가 잘 동작한다

# Good
SC-05: 노드 생성 시 타임스탬프 자동 부여, 드래그로 위치 이동 가능
```

### 정량 기준 포함
```
# Bad
SC-01: 앱이 빨리 실행된다

# Good
SC-01: 앱 실행 시간 ≤ 5초
```

## TCC 프로젝트 실적

- 27개 성공 기준 전부 PASS
- 4개 Phase 단계적 구현
- 사람은 한국어 자연어로만 요구 → AI가 REQ/SC로 구조화
- 실험 단위 추적 가능: REQ ↔ SC ↔ EXP ↔ 커밋
